# 软件总体方案：RPA 机器人部分与对账部分

## 1. 总体定位

本软件定位为本地 Windows 桌面业务工具，分为两条主线：

1. RPA 机器人部分：先把 Excel 数据导入本软件系统，再控制电脑上的 EXE 软件，根据系统内待处理数据完成查询、录入、保存等操作；执行成功后更新系统业务数据，并按需导出 Excel 结果。
2. 对账部分：读取药师帮对账单 Excel 和入库系统 MySQL 数据，执行商品级入库对账，并导出对账结果 Excel。

软件不是一次性脚本，而是一个可持续扩展的平台。两部分共用任务中心、Excel 处理、配置管理、日志、截图、执行记录、结果导出等基础能力。

## 2. 软件模块划分

建议主界面按以下模块设计：

1. 首页 / 任务中心
2. RPA 机器人
3. 入库对账
4. 执行记录
5. 配置中心
6. 日志与截图

其中：

- RPA 机器人：面向 EXE 软件自动化操作。
- 入库对账：面向药师帮账单与入库系统数据核对。
- 执行记录：统一记录 RPA 任务和对账任务。
- 配置中心：统一维护 EXE 软件配置、Excel 模板、数据库配置、SQL 配置、对账规则。

## 3. RPA 机器人部分

RPA 机器人专项细化方案见：

[RPA机器人专项设计方案.md](./RPA机器人专项设计方案.md)

### 3.1 功能目标

RPA 机器人用于控制电脑上的 EXE 软件。Excel 数据先导入本软件系统，机器人再根据系统内待处理数据自动完成业务操作。

第一版目标：

- 读取并导入 Excel 数据；
- 在系统中生成待处理数据；
- 连接或启动目标 EXE 软件；
- 自动执行查询、录入、保存等动作；
- 每行独立判断成功或失败；
- 将执行状态、系统单号、失败原因、目标 EXE 返回信息写回系统；
- 执行成功后更新系统业务数据；
- 按需导出 Excel 结果；
- 保存日志和失败截图；
- 支持断点续跑和失败重试。

### 3.2 适用场景

适合处理以下业务：

- 按系统内待处理数据录入单据；
- 按系统内查询条件查询目标 EXE 信息；
- 查询或录入成功后更新系统业务数据；
- 批量维护系统资料；
- 批量生成或提交业务单据。

### 3.3 用户操作流程

1. 选择 Excel 文件；
2. 选择业务模板；
3. 预览 Excel 数据；
4. 配置或确认字段映射；
5. 校验必填字段；
6. 选择执行模式：全部执行、仅执行待处理、仅重试失败；
7. 点击开始执行；
8. 软件连接目标 EXE；
9. 按系统待处理数据逐行执行查询、录入、保存；
10. 实时显示进度和日志；
11. 执行结果写回系统数据库，成功后更新系统业务数据；
12. 按需导出 Excel 结果；
13. 生成任务记录、日志和失败截图。

### 3.4 页面功能

RPA 机器人页面包含：

- Excel 文件选择；
- Sheet 选择；
- 数据预览；
- 字段映射配置；
- 业务模板选择；
- 任务模式选择；
- 开始、暂停、继续、停止按钮；
- 执行进度统计；
- 实时日志；
- 结果文件路径；
- 失败截图入口。

### 3.5 Excel 字段设计

基础字段可根据具体业务模板变化，通用字段建议包括：

- 序号；
- 单据类型；
- 查询条件；
- 客户名称；
- 商品名称；
- 数量；
- 单价；
- 金额；
- 备注。

系统写回字段统一为：

- 执行状态；
- 执行时间；
- 失败原因；
- 系统返回信息；
- 截图路径；
- 任务批次号。

执行状态固定为：

- 待处理；
- 处理中；
- 成功；
- 失败；
- 跳过。

### 3.6 EXE 控制方案

优先使用：

- pywinauto；
- uiautomation。

如果目标软件控件不可识别，再补充：

- 快捷键；
- 坐标点击；
- 图像识别；
- OCR 识别。

第一步必须做 EXE 可控性验证：

- 能否识别主窗口；
- 能否识别菜单、按钮、输入框、表格；
- 能否自动输入查询条件；
- 能否点击查询或保存；
- 能否识别系统提示。

### 3.7 稳定性设计

RPA 部分必须支持：

- 每行独立异常捕获；
- 一行失败不影响后续行；
- 失败截图；
- 操作日志；
- 超时重试；
- 软件无响应检测；
- 断点续跑；
- 防重复提交。

防重复建议：

- 录入前先查询是否已存在相同单据；
- 如果系统无法查询重复，则至少在 Excel 中记录任务批次号、系统返回编号、已提交状态。

## 4. 对账部分

### 4.1 功能目标

对账部分用于核对药师帮采购账单与入库系统实际入库记录是否一致。

核心目标：

- 找出药师帮已采购但入库系统未入库的商品；
- 找出入库系统有记录但药师帮账单中没有的商品；
- 核对商品数量、单价、金额是否一致；
- 保留订单号、供应商、批号、有效期等追溯信息；
- 输出清晰的 Excel 对账结果，方便人工处理差异。

第一版重点解决商品级对账，不自动修改入库系统数据。

### 4.2 数据来源

药师帮侧：

- 使用“我的账单”页面下载的对账单 Excel；
- 第一版采用手工下载；
- 第一版主数据使用工作表 `本月支付账单明细`。

入库系统侧：

- 使用 MySQL 数据库；
- 第一阶段使用本地 MySQL 测试库；
- 后续可切换正式只读库；
- 对账程序只读查询，不修改数据库。

### 4.3 药师帮 Excel 工作表

药师帮样表包含：

- `本月支付订单`：订单级汇总，可辅助核对订单金额、实际支付金额、退款金额，读取时排除最后的汇总行。
- `本月支付账单明细`：商品明细级数据，作为第一版对账主数据。
- `往月订单本月退款`：用于处理跨月退款，第一版可先读取和展示，是否参与对账后续确认。

### 4.4 药师帮关键字段

`本月支付账单明细` 重点读取：

- 订单号；
- 订单类型；
- 药店名称；
- 供应商；
- 企业名称；
- 采购时间；
- 商品名称；
- 厂家；
- 规格；
- 单位；
- 批准文号；
- 条形码；
- 批号；
- 生产日期；
- 有效期；
- 单价；
- 折后价；
- 下单数量；
- 退款数量；
- 折后价总额；
- 商品总金额；
- 运费；
- 优惠金额。

注意事项：

- 条形码必须按文本处理，避免科学计数法；
- 同一订单号会对应多行商品；
- 订单级金额字段可能只在同订单第一行有值；
- 退款数量是否从下单数量中扣除，需要业务确认，第一版建议默认使用净采购数量。

### 4.5 入库系统标准 SQL

对账功能不直接绑定复杂业务表结构，而是通过一段“入库明细 SQL”查询出标准结果。

第一版入库 SQL 必须至少输出：

- inbound_no：入库单号；
- inbound_date：入库日期；
- product_name：商品名称；
- manufacturer：生产厂家；
- spec：规格；
- quantity：入库数量；
- unit_price：入库单价；
- amount：入库金额。

建议尽量输出：

- supplier_code：供应商编码；
- supplier_name：供应商名称；
- product_code：商品编码；
- ysb_code：药师帮编码；
- barcode：条形码；
- batch_no：批号；
- expiry_date：有效期；
- document_status：单据状态。

### 4.6 数据库配置

配置中心需要支持 MySQL 配置：

- 配置名称；
- 数据库类型，第一版固定 MySQL；
- 主机地址；
- 端口，默认 3306；
- 数据库名；
- 用户名；
- 密码；
- 字符集，默认 utf8mb4；
- 连接超时时间；
- 是否启用；
- 入库明细 SQL。

必须支持：

- 新增、编辑、启用数据库配置；
- 测试数据库连接；
- 保存入库明细 SQL；
- 预览 SQL 前 20 行；
- 校验 SQL 输出字段；
- 对账时自动使用当前启用配置。

SQL 安全限制：

- 只允许执行 SELECT；
- 禁止 INSERT、UPDATE、DELETE、DROP、TRUNCATE、ALTER；
- 推荐使用只读数据库账号；
- 密码不在界面明文显示；
- 配置文件不提交到代码仓库。

### 4.7 对账口径

第一版主口径：

- 药师帮侧：`本月支付账单明细`；
- 入库系统侧：入库明细 SQL 查询结果；
- 时间范围：按账期选择，例如 2026-04-01 至 2026-04-30。

数量口径：

- 默认使用净采购数量 = 下单数量 - 退款数量；
- 是否扣除退款数量需要业务确认；
- 往月订单本月退款是否冲减本月对账，第一版暂不自动处理。

金额口径：

- 数量：净采购数量；
- 单价：折后价；
- 金额：折后价总额；
- 订单级支付金额、运费、优惠只作为辅助汇总，不直接参与商品级入库金额核对。

时间口径：

- 药师帮侧是采购时间；
- 入库系统侧是入库日期；
- 入库查询日期范围允许比采购账期适当放宽，例如 4 月采购账期可查询 4 月 1 日至 5 月 10 日。

### 4.8 匹配规则

第一版匹配优先级：

1. 条形码一致；
2. 商品编码或药师帮编码一致；
3. 商品名称 + 规格 + 厂家一致；
4. 无法唯一匹配时标记为“需要人工确认”。

多行聚合维度建议：

- 商品匹配键；
- 供应商；
- 规格；
- 厂家。

是否加入批号、有效期作为强匹配维度，需要根据入库系统数据质量确认。

### 4.9 核对规则

第一版核对规则：

- 数量差异 = 入库数量 - 药师帮净采购数量；
- 单价差异 = 入库单价 - 药师帮折后价；
- 金额差异 = 入库金额 - 药师帮折后价总额；
- 单价误差默认 0.01 元以内视为一致；
- 金额误差默认 0.01 元以内视为一致；
- 数量必须完全一致，除非后续确认允许误差。

差异类型：

- 一致；
- 入库系统缺少；
- 药师帮缺少；
- 数量不一致；
- 单价不一致；
- 金额不一致；
- 商品信息不一致；
- 供应商不一致；
- 批号或有效期不一致；
- 疑似重复记录；
- 需要人工确认。

一个商品可以同时命中多个差异。

### 4.10 输出结果

第一版输出 Excel 对账结果，至少包含：

1. `汇总`
2. `对账明细`
3. `差异明细`

汇总页展示：

- 药师帮账期；
- 入库查询日期范围；
- 药师帮商品行数；
- 入库系统商品行数；
- 一致品种数；
- 差异品种数；
- 药师帮总数量；
- 入库总数量；
- 药师帮总金额；
- 入库总金额；
- 金额差异；
- 各差异类型数量。

对账明细页展示：

- 对账状态；
- 匹配方式；
- 药师帮订单号；
- 入库单号；
- 药师帮供应商；
- 入库供应商；
- 商品名称；
- 规格；
- 厂家；
- 条形码；
- 批号；
- 有效期；
- 药师帮数量；
- 入库数量；
- 数量差异；
- 药师帮单价；
- 入库单价；
- 单价差异；
- 药师帮金额；
- 入库金额；
- 金额差异；
- 备注。

差异明细页只展示不一致或需要人工确认的记录，用于人工补入库、查漏单、查价格、查供应商、查商品映射。

可选保留原始数据页：

- 药师帮原始明细；
- 入库 SQL 查询结果；
- 往月退款数据。

- 自动修改入库系统数据库；
- 自动生成入库单；
- 复杂审批流；
- 正式库密钥加密管理；
- 多用户权限系统；

## 9. 验收标准

RPA 机器人部分：

- 能读取指定 Excel；
- 能识别必填字段；
- 能连接或启动目标 EXE；
- 能完成一次查询或录入验证；
- 能按行写回成功或失败；
- 失败时能记录原因和截图；
- 中断后可继续处理待处理或失败行。

对账部分：

- 能成功读取样表 `15861655561-2026-05-15.xlsx`；
- 能识别 `本月支付账单明细`；
- 能连接本地 MySQL；
- 能执行只读入库明细 SQL；
- SQL 预览能显示前 20 行；
- 缺少必要字段时能提示；
- 能生成对账结果 Excel；
- 结果 Excel 至少包含 `汇总`、`对账明细`、`差异明细`；
- 差异行能明显识别；
- 不对入库系统数据库执行任何写操作。

## 10. 待确认问题

RPA 机器人部分需要确认：

1. 目标 EXE 软件名称和安装路径；
2. 是否需要机器人自动登录；
3. Excel 样例文件和字段；
4. 人工操作流程截图或录屏；
5. 成功、失败、重复提交的判断规则；
6. 是否需要多套业务模板。

对账部分需要确认：

1. 入库系统 MySQL 表结构；
2. 入库系统是否有只读账号；
3. 入库系统是否保存条形码；
4. 入库系统是否保存药师帮编码；
5. 药师帮退款数量是否要从下单数量中扣除；
6. 往月订单本月退款是否参与本月对账；
7. 入库查询日期需要比采购账期放宽几天；
8. 单价差异允许误差是多少；
9. 金额差异允许误差是多少；
10. 供应商不一致是否算严重差异；
11. 是否需要按批号和有效期精确核对；
12. 是否需要维护商品映射表。
