# 企业微信饮食记录机器人 · 完整方案

## 1. 业务概述

**目标**：通过企业微信定时提醒顾客上传三餐照片，自动识别餐食中碳水、蔬菜、肉类的占比，并通知对应服务人员进行点评。若服务人员超时未点评，自动逐级升级通知（服务人员 → 上级 → BOSS）。

**核心流程**：

```
定时器触发 ──▶ 发送提醒通知 ──▶ 顾客拍照上传 ──▶ AI识别分析 ──▶ 通知服务人员点评
（早7/午12/晚18）  （企业微信）     （群聊发图）    （GPT-4o Vision）        │
                                                                          │
                                                    ┌─────────────────────┘
                                                    ▼
                                        ┌─────────────────────────┐
                                        │  5分钟内点评？           │
                                        └──────┬──────────┬───────┘
                                          是    │          │  否
                                           ▼    │          ▼
                                       记录完毕  │  通知服务人员上级
                                               │          │
                                               │  ┌───────┴────────┐
                                               │  │ 10分钟内回应？   │
                                               │  └───────┬────────┘
                                               │     否   │
                                               │          ▼
                                               │  通知 BOSS
                                               │          │
                                               └──────────┘
```

### 4.2 crypto.py — 企业微信加解密

实现标准 `WXBizMsgCrypt` 协议：

**URL验证（GET请求）**：
```
企业微信请求: GET /wecom/callback?msg_signature=xxx&timestamp=xxx&nonce=xxx&echostr=xxx

后端处理:
1. 计算签名: SHA1(token, timestamp, nonce, echostr_raw) → 与 msg_signature 比对
2. AES-256-CBC 解密 echostr → 提取明文
3. 返回明文 echostr（URL验证通过）
```

**消息接收（POST请求）**：
```
企业微信推送: POST /wecom/callback (XML body)

后端处理:
1. 验证 msg_signature
2. AES解密 Encrypt 字段
3. 解析解密后的XML → 获取消息类型、内容、发送者、MediaId等
```

**加密规格**：
- 算法：AES-256-CBC，PKCS7填充
- Key：EncodingAESKey Base64解码后取前32字节
- IV：Key的前16字节
- 明文结构：`[16随机字节][4字节网络序消息长度][消息XML][企业ID]`

### 4.3 server.py — FastAPI回调服务

**端点列表**：

| 方法 | 路径 | 说明 | 认证 |
|------|------|------|------|
| GET | `/wecom/callback` | 企业微信URL验证 | 签名验证 |
| POST | `/wecom/callback` | 接收企业微信消息推送 | 签名验证 |
| POST | `/api/wecom/feedback` | 服务员提交点评 | 签名验证 |
| GET | `/health` | 健康检查 | 无 |
| GET | `/records` | 查询分析记录 | 无（内部） |
| GET | `/records/stats` | 统计报表 | 无（内部） |

**POST回调处理流程（图片消息）**：
```
1. 验签 → 2. 解密XML → 3. 判断消息类型：

   图片消息(msg_type=image):
   │
   ├─ 1. 提取 MediaId
   ├─ 2. 通过企业微信素材API下载图片
   ├─ 3. 调用 food_analyzer 分析（GPT-4o Vision）
   ├─ 4. 保存分析结果到数据库
   ├─ 5. 发送分析结果给顾客（群聊）
   ├─ 6. 查询该顾客绑定的服务人员
   ├─ 7. 通知服务人员进行点评（应用消息）
   ├─ 8. 启动点评超时计时器（5分钟）
   │
   └── 文本消息(msg_type=text):
       1. 提取文本内容
       2. 判断是否为服务人员的点评回复
       3. 若是：取消超时计时器，保存点评内容
       4. 若否：简单回复或忽略
```

### 4.4 scheduler.py — 定时调度器

使用 `APScheduler` 实现定时任务：

| 任务名称 | 触发时间 | 内容 |
|----------|----------|------|
| `breakfast_reminder` | 每日 07:00 | "早上好！请上传您的早餐照片 🍳" |
| `lunch_reminder` | 每日 12:00 | "中午好！请上传您的午餐照片 🥘" |
| `dinner_reminder` | 每日 18:00 | "晚上好！请上传您的晚餐照片 🍜" |
| `summary_report` | 每日 21:00 | 汇总当日三餐分析结果 |
| `check_escalations` | 每 1 分钟 | 检查是否有待处理的点评超时升级 |

**通知方式**：调用企业微信API → `POST /cgi-bin/message/send` → 发送文本消息到群聊

**消息模板**：
```
📢 饮食记录提醒
━━━━━━━━━━━━━━━━
大家好！
请拍摄并发送您今天的{餐次}照片。
发送图片后，我会自动分析您的饮食结构。
━━━━━━━━━━━━━━━━
```

**后台上报接口 `/api/wecom/feedback`**：
```
服务人员通过企业微信发送文本消息点评，格式:
  @顾客姓名 点评内容

示例：
  @张三 今天碳水摄入偏高，建议晚餐减少主食
```

### 4.7 escalation.py — 点评超时升级引擎

这是核心逻辑模块，管理点评超时检测和逐级升级通知。

**状态机**：

```
                    ┌──────────────────┐
                    │  等待服务人员点评  │
                    │  (等待5分钟)      │
                    └────────┬─────────┘
                             │
                    ┌────────┴────────┐
                    │ 是否5分钟内点评？ │
                    └────────┬────────┘
                       是    │    否
                        ▼    │    ▼
                 ┌──────────┐│ ┌──────────────────┐
                 │ 点评完成  ││ │  通知上级         │
                 │ 流程结束  ││ │  (等待10分钟)     │
                 └──────────┘│ └────────┬─────────┘
                             │          │
                             │ ┌────────┴────────┐
                             │ │ 是否10分钟内回应？│
                             │ └────────┬────────┘
                             │     是    │    否
                             │     ▼     │    ▼
                             │ ┌───────┐ │ ┌──────────┐
                             │ │流程结束│ │ │ 通知BOSS  │
                             │ └───────┘ │ └──────────┘
                             │           │     │
                             └───────────┘     ▼
                                         ┌──────────┐
                                         │ 流程结束  │
                                         └──────────┘
```

**数据结构**：
```python
@dataclass
class EscalationTask:
    record_id: int              # 分析记录ID
    user_id: str                # 顾客企业微信UserID
    staff_user_id: str          # 服务人员企业微信UserID
    superior_user_id: str       # 上级企业微信UserID
    boss_user_id: str           # BOSS企业微信UserID
    status: str                 # waiting_staff / escalated_to_superior / escalated_to_boss / completed
    notified_at: datetime       # 通知服务人员时间
    escalated_at: datetime      # 升级通知时间（可选）
```

**核心方法**：

| 方法 | 说明 |
|------|------|
| `start_escalation()` | 开始点评超时计时（分析完成时调用） |
| `check_timeouts()` | 每分钟执行，检查是否有超时 |
| `handle_staff_reply()` | 处理服务人员的点评回复 |
| `cancel_escalation()` | 取消超时升级（点评完成时调用） |

**升级通知模板**：

```

// 第二级 → 上级（5分钟超时）

━━━━━━━━━━━━━━━━
```

### 4.8 models.py — 数据模型

使用 SQLite（项目已有 `cryptography` 依赖，新增 SQLite 表）。

**customer_staff_binding 表**（顾客-服务员绑定）：
```sql
CREATE TABLE customer_staff_binding (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    customer_user_id TEXT NOT NULL UNIQUE,    -- 顾客企业微信UserID
    customer_name TEXT NOT NULL,              -- 顾客姓名
    staff_user_id TEXT NOT NULL,              -- 服务人员企业微信UserID
    staff_name TEXT NOT NULL,                 -- 服务人员姓名
    superior_user_id TEXT,                    -- 上级企业微信UserID
    superior_name TEXT,                       -- 上级姓名
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

**analysis_records 表**（分析记录）：
```sql
CREATE TABLE analysis_records (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    customer_user_id TEXT NOT NULL,           -- 顾客企业微信UserID
    customer_name TEXT NOT NULL,              -- 顾客姓名
    meal_type TEXT NOT NULL,                  -- 餐次（breakfast/lunch/dinner）
    record_date TEXT NOT NULL,                -- 记录日期（YYYY-MM-DD）
    image_url TEXT,                           -- 图片本地路径
    carb_proportion REAL,                     -- 碳水占比
    vegetable_proportion REAL,                -- 蔬菜占比
    meat_proportion REAL,                     -- 肉类占比
    other_proportion REAL,                    -- 其他占比
    total_calories INTEGER,                   -- 预估总热量
    nutrition_advice TEXT,                    -- 营养建议
    raw_analysis TEXT,                        -- 原始AI返回JSON
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

**staff_reviews 表**（服务人员点评）：
```sql
CREATE TABLE staff_reviews (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    record_id INTEGER NOT NULL,               -- 关联分析记录ID
    staff_user_id TEXT NOT NULL,              -- 服务人员UserID
    staff_name TEXT NOT NULL,                 -- 服务人员姓名
    review_content TEXT NOT NULL,             -- 点评内容
    review_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (record_id) REFERENCES analysis_records(id)
);
```

**escalation_tasks 表**（超时升级任务）：
```sql
CREATE TABLE escalation_tasks (

    customer_name TEXT NOT NULL,              -- 顾客姓名
    staff_user_id TEXT NOT NULL,              -- 服务人员UserID
    staff_name TEXT NOT NULL,                 -- 服务人员姓名
    superior_user_id TEXT,                    -- 上级UserID
    superior_name TEXT,                       -- 上级姓名
    boss_user_id TEXT,                        -- BOSS UserID
    boss_name TEXT,                           -- BOSS姓名
    status TEXT DEFAULT 'waiting_staff',      -- waiting_staff / escalated_to_superior / escalated_to_boss / completed
    staff_notified_at DATETIME,               -- 通知服务人员时间
    superior_notified_at DATETIME,            -- 通知上级时间
    boss_notified_at DATETIME,                -- 通知BOSS时间
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (record_id) REFERENCES analysis_records(id)
```

**meal_reminders 表**（定时提醒配置）：

## 5. 完整交互流程示例

### 场景：午餐提醒 + 服务员点评 + 超时升级

```
时间   事件
────  ───────────────────────────────────────────────────────────

12:00  调度器触发 → 群发通知
       📢 饮食记录提醒
       大家好！午餐时间到啦 🥘
       请拍摄并发送今天的午餐照片。
       ━━━━━━━━━━━━━━━━━━━━━

12:05  顾客A发送午餐照片到群聊
       后端处理：
       1. 回调接收图片消息
       2. GPT-4o Vision分析
       3. 保存分析结果
       4. 群聊发送分析结果
       5. 查询顾客A绑定的服务员A
       6. 应用消息通知服务员A：
          → 📝 顾客A已上传午餐照片，请及时点评

12:10  服务员A在群聊发送点评（5分钟内完成）
       @顾客A 今天午餐的蛋白质摄入不错，
       建议增加一些蔬菜。
       
       后端处理：
       1. 回调接收文本消息
       2. 匹配到是点评回复
       3. 保存点评到 staff_reviews
       4. 取消超时升级任务
       5. 流程结束 ✅
```

## 6. 部署方案

### 6.1 公网访问

企业微信回调URL必须是公网可达的，有两种方式：

| 方式 | 优点 | 缺点 |
|------|------|------|
| **云服务器部署** | 稳定可靠，正式使用 | 需要服务器费用 |
| **内网穿透(frp/ngrok)** | 免费，开发测试方便 | 不稳定，有带宽限制 |

**推荐**：开发用 frp/ngrok，上线用云服务器。

### 6.2 企业微信后台配置

在 [企业管理后台](https://work.weixin.qq.com/wework_admin/frame#apps) 创建自建应用：

```
1. 我的企业 → 企业信息 → 复制 CorpID
2. 应用管理 → 创建应用 → 获取 AgentId 和 Secret
3. 应用 → 设置 → 配置回调:
   - URL: https://你的域名/wecom/callback
   - Token: 自定义（与配置一致）
   - EncodingAESKey: 随机生成（与配置一致）
4. 应用 → 设置 → 配置接收消息:
   - 勾选"图片消息"、"文本消息"
5. 应用 → 发送消息 → 配置可以发送到群聊及指定用户
```

### 6.3 首次使用初始化

1. 启动服务，完成企业微信回调验证
2. 通过 API 或手动插入 `customer_staff_binding` 数据（顾客-服务员绑定关系）
3. 通过 API 或手动插入 `meal_reminders` 数据（定时提醒配置）
4. 配置 `WECOM_BOSS_USERID` 环境变量

**管理员绑定接口（用于后台配置）**：
```json
POST /api/admin/bind
{
  "customer_user_id": "zhangsan",
  "customer_name": "张三",
  "staff_user_id": "lisi",
  "staff_name": "李四",
  "superior_user_id": "wangwu",
  "superior_name": "王五"
}
```

### 6.4 启动方式

```bash
# 安装依赖
pip install fastapi uvicorn apscheduler openai httpx

# 启动服务
set WECOM_CORPID=你的企业ID
set WECOM_CORPSECRET=你的应用Secret
set WECOM_AGENTID=1000001
set WECOM_TOKEN=你的Token
set WECOM_ENCODING_AES_KEY=你的AESKey
set OPENAI_API_KEY=sk-你的API密钥
set WECOM_BOSS_USERID=boss的企业微信ID

python -m app.wecom_bot.run
```

## 8. 注意事项

1. **Token安全**：OpenAI API Key、企业微信 Secret 等敏感信息不要硬编码在代码中，用环境变量或 `.env` 文件管理
2. **并发处理**：群聊多人同时发图时，回调可能并发，需处理好异步任务队列
3. **图片存储**：企业微信素材API的图片有效期3天，需及时下载并转存
4. **错误处理**：AI识别失败、网络超时等异常需有重试机制和降级方案
5. **频率限制**：企业微信API有调用频率限制（access_token 2000次/小时，消息发送频率限制）
6. **代理兼容**：当前环境 `HTTPS_PROXY=http://127.0.0.1:7897`，代码中需兼容代理设置
7. **顾客-服务员绑定**：首次使用需要在后台手动绑定关系，建议提供管理端界面
8. **消息ID去重**：企业微信回调可能重复推送同一消息，需用 `MsgId` 做去重处理
