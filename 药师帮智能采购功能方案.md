# 药师帮智能采购功能方案

## 1. 功能目标

将门店采购目录中的品种，按既定采购规则在药师帮逐个匹配、逐个加购、购物车验证和结果回填。

本功能归属于软件一级菜单 `智能采购`，当前入口为：

```text
智能采购
└─ 药师帮智能采购
```

后续新增的采购相关功能也统一放在 `智能采购` 菜单下面，不再分散到 `RPA机器人`、`药师帮对账` 或其他模块中。

核心目标：

```text
以表格目录为采购依据
以本次确认的供应商范围为下单范围
以期望价格规则控制价格风险
以购物车实际验证作为采购成功标准
成功和失败都回填原因
```

## 2. 输入资料

每次执行前需要确认：

```text
采购目录 Excel
本次处理的工作表名称
本次允许采购的供应商范围
是否允许购物车保留原有商品
是否允许团购/包邮凑单例外
```

采购目录建议字段：

```text
商品编码
商品名称
规格
生产厂家
批准文号
采购数量
期望价格
药师帮商品名称
药师帮规格
药师帮批准文号
商家
企业名称
起购数量
可用库存
实际采购数量
药师帮采购价
是否选中
活动类型
```

## 3. 供应商范围

每次采购必须重新确认供应商范围，不默认沿用历史范围。

供应商需要同时支持显示名和企业全称匹配，例如：

```text
小药精 = 江苏金石医药有限公司
采药易 = 广东东健医药有限公司
药中缘 = 河南惠盈医药有限公司
123易购[安通医药] = 重庆安通医药有限公司
```

常规采购必须在本次确认的供应商范围内。建议团购、包邮凑单品种如用户允许，可不受供应商范围限制，但必须单独标记原因。

## 4. 采购价格规则

药师帮候选价格需要先按折扣口径折算，再与期望价格上限比较：

```text
折算采购价 = 药师帮候选采购价 * 0.97
```

折算采购价不得超过以下两个上限中的较低值：

```text
期望价格 * 1.05
期望价格 + 1
```

也就是：

```text
折算采购价 <= 期望价格 * 1.05
且
折算采购价 <= 期望价格 + 1
```

任何一个条件不满足，都记录为：

```text
供应商范围内候选价格超过期望价5%或1元
```

价格列必须使用实际采购价格或购物车价格，不得使用期望价格代替采购价格。结果回填时保留购物车真实采购价，规则判断时记录折算采购价和价格判断依据。

## 5. 商品匹配规则

优先匹配顺序：

1. 批准文号一致。
2. 商品名称/通用名一致。
3. 规格一致或符合单位容差。
4. 厂家一致或高度相似。
5. 供应商在本次范围内。
6. 价格、起购数量、库存满足规则。

智能采购匹配结果只作为参考，不直接作为最终下单结果。系统需要逐个品种搜索候选，按规则匹配；匹配成功后立即加入购物车并验证，匹配不成功则直接回写失败原因并继续下一品种。

## 5.2 名称/批准文号/厂家容差匹配

商品匹配允许做分层容差，但必须遵循“批准文号优先、厂家辅助、名称兜底”的原则。

### 5.2.1 批准文号容差

批准文号是最强匹配依据。

允许容差：

```text
忽略空格
忽略大小写
忽略全角/半角差异
忽略“国药准字”前后多余符号
```

示例：

```text
国药准字Z11020152
国药准字 Z11020152
Z11020152
```

以上可视为同一批准文号。

不允许容差：

```text
批准文号主体字母或数字不同
H/Z/S/B 等类别字母不同
批准文号缺失且名称厂家规格也无法确认
```

### 5.2.2 厂家容差

厂家匹配允许做主体名容差。

可忽略的企业后缀：

```text
有限公司
有限责任公司
股份有限公司
集团
药业集团
医药集团
制药厂
制药有限公司
药业有限公司
医药有限公司
```

示例：

```text
北京同仁堂科技发展股份有限公司制药厂
北京同仁堂科技发展股份有限公司
北京同仁堂科技发展制药厂
```

可以视为同一厂家主体。

厂家容差判断：

```text
主体名完全包含
主体名双字片段相似度达到阈值
品牌/集团主体一致且批准文号一致
```

厂家不允许容差：

```text
批准文号不同且厂家主体不同
仅品牌相同但生产厂家明显不同
委托生产关系未在页面或表格中体现
```

### 5.2.3 名称容差

名称匹配允许去除品牌、括号、营销词、活动词后比较通用名。

可忽略内容：

```text
括号内品牌名
【首推】
包邮
拼团
起购
折后
日常特卖
乐药师
厂家简称
仓库后缀
```

示例：

```text
知柏地黄丸(同仁堂)
【央拓医药】同仁堂 知柏地黄丸360丸
```

可提取通用名为：

```text
知柏地黄丸
```

名称容差不能单独决定采购，至少还需要满足以下任一强条件：

```text
批准文号一致
或 厂家主体高度一致 + 规格一致
或 批准文号一致且名称/规格/厂家无冲突
```

### 5.2.4 匹配置信度

建议按置信度分级：

```text
高置信：批准文号一致 + 规格一致
中置信：厂家主体一致 + 名称通用名一致 + 规格一致
低置信：仅名称相似
```

执行策略：

```text
高置信：可自动采购
中置信：价格和供应商合规时可采购，但需记录匹配依据
低置信：不自动采购，记录需人工确认
```

### 5.2.5 药师帮编码定位

药师帮编码只作为辅助标识和购物车反写匹配依据，不作为采购优先选品依据。

采购查询关键词：

```text
优先使用药师帮商品名称
没有药师帮商品名称时使用采购表商品名称
```

采购选品仍按名称、规格、厂家、供应商、价格、起购数量、库存综合判断。不得因为药师帮编码一致而绕过名称、规格、厂家校验。

## 6. 规格单位与包装总数规则

规格判断以“强度 + 包装总数”为核心。包装/剂型计数单位可以容差，但必须计算包装总数，不能只看局部数字。

可容差单位：

```text
粒
丸
片
板
袋
支
瓶
贴
枚
盒
包
只
s
```

示例：

```text
360粒 可视为 360丸
20片*3板 可视为 60片
0.6g*36s 可视为 0.6g*12片*3板
75mg*28片 可视为 75mg*7片*4板
```

以下情况必须判定为规格冲突，不能自动采购，也不能购物车反写匹配：

```text
80mg*7粒 不能视为 80mg*28粒
75mg*7片 不能视为 75mg*7片*4板
强度一致但包装总数不同，必须硬拦截
```

不可容差单位：

```text
mg
g
ml
ug/μg
%
克
毫升
```

示例：

```text
2.5mg 不能视为 5mg
10ml 不能视为 100ml
```

## 7. 购物车处理规则

执行前先检查登录状态。若页面跳转登录、出现验证码、风控、账号异常，应停止执行。

如果用户明确要求清空购物车：

```text
每次只删除一个商品
删除后刷新购物车验证
确认消失后再删除下一个
```

如果用户要求购物车不要清空：

```text
加购前先扫描购物车
购物车已有同品种且数量满足要求，不重复加购
购物车已有同品种但数量不足，记录并等待人工确认
```

## 8. 加购流程

标准流程：

```text
读取 Excel 目录
确认供应商范围
检查登录状态
扫描当前购物车
逐个品种搜索
提取药师帮候选商品
按供应商、规格、厂家/批准文号、价格、起购、库存筛选
匹配成功：选择合规候选并调用加购接口
匹配成功：进入购物车验证编码、品种、数量、价格
匹配成功：写入成功结果
匹配失败：直接写入失败原因
继续下一个品种
```

加购必须逐个执行。一个商品成功加入并验证后，再处理下一个商品。

## 9. 成功标准

只有满足以下条件才记录为成功：

```text
接口返回加购成功或购物车已存在
购物车中能找到对应商品
购物车数量满足采购数量
供应商和价格可追溯
```

成功原因建议：

```text
已加购并验证
购物车已存在，未重复加购
```

## 10. 失败原因标准化

失败原因应尽量标准化，便于后续统计和优化：

```text
登录状态异常
未找到满足供应商、品种、规格、厂家/批准文号、价格、起购数量的候选
供应商范围内候选价格超过期望价5%或1元
供应商范围内候选起购数量大于采购数量
库存不足
拼团商品无法加入购物车
加购后未验证成功
购物车已有但数量不足，需人工确认
候选搜索超时
```

## 11. 回填结果

采购完成后生成结果表，至少回填：

```text
采购状态
采购供应商
采购品种
采购规格
采购厂家
采购数量
采购价格
药师帮编码
最高允许价
采购原因
```

同时生成汇总页：

```text
目录行数
已处理行数
成功行数
未成功行数
失败原因分布
```

## 12. 二次重试规则

对失败原因为：

```text
未找到满足供应商、品种、规格、厂家/批准文号、价格、起购数量的候选
```

可进行二次重试。

二次重试前应先检查：

```text
是否存在规格单位容差问题
搜索页是否加载完成
候选列表是否提取为空
是否需要使用药师帮商品名称、批准文号或更稳定的规格关键词辅助搜索
```

二次重试只处理该类失败行，不重复处理已成功、价格超限、起购量超限、拼团无法加购物车的行。

## 13. 风控与安全

为降低账号异常风险：

```text
逐个加购
每个品种之间保留等待间隔
加购前后都进行页面验证
出现登录异常、验证码、风控提示立即停止
不要高速批量点击
不要重复加入购物车
```

## 14. 当前待优化点

1. 搜索页加载完成判断需要更稳定，避免候选列表仍在 Loading 时误判。
2. 对“单位容差”和包装总数规则需要底表化维护，例如 `丸/粒`、`s/片`、`袋/包`、`7片*4板=28片`。
3. 对失败原因为“未找到候选”的记录，应保留候选快照，方便人工复核。
4. 对购物车反写过程应保存购物车快照和匹配明细，方便追溯错配原因。
5. 对团购/包邮商品可单独形成“建议清单”，不直接替代常规采购。

## 15. 执行原则

```text
先确认供应商范围
先检查登录状态
不清空购物车，除非用户明确要求
不重复加购已存在商品
不采购价格异常商品
不采购起购数量不满足商品
不绕过拼团商品限制
所有结果必须回填原因
```

# 技术方案

## 16. 总体架构

智能采购能力拆分为 6 个模块：

```text
Excel 解析模块
采购规则引擎
药师帮页面/接口适配模块
购物车状态模块
采购执行模块
结果回填与审计模块
```

整体数据流：

```text
采购目录 Excel
-> 标准采购任务 JSON
-> 当前购物车快照
-> 药师帮候选商品列表
-> 规则引擎筛选
-> 单品加购
-> 购物车验证
-> 结果 JSON
-> 回填 Excel
```

## 17. 模块职责

### 17.1 Excel 解析模块

职责：

```text
读取 .xls/.xlsx
识别工作表
标准化字段名
提取采购行
生成采购任务 JSON
```

### 17.2 采购规则引擎

职责：

```text
供应商范围校验
名称/批准文号/厂家匹配
规格单位容差匹配
价格阈值校验
起购数量校验
库存校验
候选排序
失败原因归类
```

候选排序建议：

```text
批准文号一致优先
厂家相似度高优先
价格低优先
起购数量低优先
有购物车加购能力优先
非拼团商品优先
```

匹配评分建议：

```text
名称权重：0.62
规格权重：0.20
厂家权重：0.18
采购最低通过分：70
自动通过阈值：85
疑似匹配阈值：70
购物车反写最低分：62
规格包装总数冲突：硬拦截
```

自动采购建议阈值：

```text
分数 >= 85：自动通过候选
分数 70-84：满足供应商、规格、厂家、价格、起购、库存规则时可采购，并记录匹配依据
分数 < 70：不自动采购，记录失败原因
```

匹配归因字段：

```text
批准文号一致
通用名一致
规格单位容差
包装总数一致
包装总数冲突
厂家主体一致
厂家主体相似
供应商范围通过
价格规则通过
低置信需人工确认
```

### 17.3 药师帮页面/接口适配模块

职责：

```text
检查登录状态
进入搜索页
等待搜索结果加载完成
提取 drugList 候选列表
调用加入购物车接口
识别接口错误
识别风控/验证码/登录异常
```

页面候选提取优先级：

```text
优先读取 Vue 组件 drugList
其次读取页面可见商品卡片
最后使用药师帮商品名称或采购表商品名称重新构造搜索关键词
```

加购接口语义：

```text
接口成功不等于最终成功
必须进入购物车再次验证
```

### 17.4 购物车状态模块

职责：

```text
读取当前购物车
识别真实商品行
过滤商家分组/无效节点
检查是否重复
检查是否已存在同品种
验证加购后的商品、数量、价格
```

购物车商品结构：

```json
{
  "wholesaleId": 244969050,
  "name": "【首推】乐药师 左炔诺孕酮片 1.5mg*1片",
  "spec": "1.5mg*1片",
  "maker": "北京法莫斯达制药科技有限公司",
  "supplier": "小药精",
  "amount": 3,
  "price": 6.18
}
```

真实商品判定：

```text
有 wholesaleId
有商品名称
amount > 0
price > 0
```

## 18. 执行状态机

每一行采购任务应有独立状态：

```text
pending
checking_cart
searching
matching
price_checking
adding
verifying
success
failed
stopped
```

状态说明：

```text
pending：待处理
checking_cart：检查购物车是否已有
searching：搜索候选商品
matching：执行规则匹配
price_checking：校验价格规则
adding：调用加购
verifying：购物车验证
success：成功
failed：该行失败但可继续下一行
stopped：登录、风控、系统异常，需要整体停止
```

## 20. 断点续跑机制

执行过程必须持续写入结果文件。

断点续跑规则：

```text
结果文件中已有 success 的行不再重复处理
结果文件中已有 failed 的行默认不重复处理
用户指定重试某类失败原因时，只抽取该类行重跑
同一行多次结果以 success 优先，其次取最后一次有效失败
```

二次重试输入：

```text
从结果 JSON 中筛选指定失败原因
回查原始采购任务
生成 retry_candidates.json
```

## 24. 风控控制策略

药师帮采购属于真实账号操作，必须采用低频、可验证执行。

建议：

```text
逐个品种处理
每个品种之间等待 5-8 秒
不要并发搜索或并发加购
不要频繁刷新同一商品
不要重复提交同一 wholesaleId
出现异常立即停止
```

高风险信号：

```text
跳转登录页
验证码
接口连续失败
购物车数量异常
页面长时间 Loading
账号提示暂停/限制
```

## 27. 评分规则表与数据落库设计

近期连续出现 `36s`、`12片*3板`、`7片*4板`、`80mg*7粒` 与 `80mg*28粒` 等边界问题，说明采购规则不能继续全部写死在代码中。

本阶段不切换 MySQL，继续使用 SQLite。当前主要问题不是数据库引擎性能，而是规则、候选、购物车快照和反写过程没有结构化落库。新表设计应避免 SQLite 专属写法，为后续迁移 MySQL 预留空间。

### 27.13 规则加载流程

```text
启动采购任务
读取默认 rule_set
加载单位别名、规格解析规则、阈值和别名
生成规则配置 JSON
Python 服务层使用该配置进行预处理和反写评分
Node 适配器通过输入 JSON 获取规则配置
若规则表为空或读取失败，使用代码内置默认规则
```

规则修改生效范围：

```text
只对新开始的采购任务生效
执行中的任务继续使用启动时加载的规则快照
历史结果不自动重算
```

### 27.14 验收用例

必须通过以下用例：

```text
养心氏片（薄膜衣片）：0.6g*36s 应匹配 0.6g*12片*3板
代文 缬沙坦胶囊80mg*7粒 不应反写到 80mg*28粒
代文 缬沙坦胶囊80mg*28粒 应反写到 80mg*28粒

泰嘉 硫酸氢氯吡格雷片75mg*7片*4板 应匹配 75mg*7片*4板
缺少药师帮编码时，仍能按名称、规格、厂家采购
药师帮编码存在但名称、规格、厂家不通过时，不允许凭编码采购
购物车已有同品种时，应反写购物车真实供应商、价格、规格、厂家、数量
匹配失败时，只写失败原因，不写采购商品、供应商、价格
```

### 27.15 发布策略

建议分两步发布：

```text
第一步：只建表、写默认规则、记录候选和购物车快照，不改变现有评分结果
第二步：逐步让采购评分和反写评分读取规则表
```

这样可以降低风险，并便于对比代码规则和表规则的结果差异。
